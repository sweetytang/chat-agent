# Message Queues

## 1. 消息队列的原理

消息队列可以把它理解成“聊天输入框前面多了一个收件箱”。

平时的聊天应用只有一条执行通道:

1. 用户发一条消息
2. 模型开始处理
3. 在这条回复结束前，下一条消息通常不能继续发

这会带来一个问题: 用户脑子里的问题往往是连续冒出来的，但系统只允许“一问一答，等前一道题做完再说”。

消息队列解决的就是这个摩擦:

1. 当前消息还在处理中时，新的消息先不要丢，也不要报错
2. 而是把它放进队列里
3. 当前任务结束后，按照先进先出的顺序继续处理下一条

LangChain 官方教程里的做法，本质上分成两层:

1. 提交层: 当前 run 正在执行时，新的输入以 `enqueue` 策略创建为待执行 run
2. 消费层: 当前 run 完成后，再把这些待执行 run 一个个接回前端流里展示

这样做的价值很直接:

1. 用户可以连续提问，不被当前回复“锁住”
2. 队列是可见的，用户知道还有哪些消息在等
3. 队列是可控的，用户可以取消某一条，或者一键清空

## 2. 结合当前项目，应该怎么改

当前 `message-queues` 子项目不是直接使用官方 `langgraph dev` Server，而是基于 Express 自己实现了一套兼容 `useStream` 的聊天链路，并且已经叠加了:

1. 线程切换
2. 分支对话
3. 人工审核中断
4. 深度思考 / 结构化输出偏好

所以这里不能粗暴照搬官方 Demo，而要顺着现有架构改。

### 当前代码里的关键差异

现状更接近“单槽位命令”:

1. `streamStore` 里只有一个 `pendingCommand`
2. 输入框在流式回复时会被禁用
3. 用户无法连续输入多条消息

这说明项目虽然叫 `message-queues`，但实际还没有完整的消息排队体验。

### 这次改造方案

为了先在当前工程里稳定落地，我们采用“前端调度队列 + 现有流式 worker 串行消费”的方式:

1. 把 `pendingCommand` 升级为 `queuedCommands[]`
2. 保留一个线程只跑一个流任务，但允许后续消息先排队
3. 当前流结束后，由 `ThreadStreamWorker` 自动消费下一条
4. 在 UI 上新增 `QueuePanel`，展示顺序、预览、单条移除、清空队列
5. 输入框在生成中不再禁用，继续发送会进入队列

这版实现和官方教程的用户体验是一致的:

1. 连续输入
2. 顺序执行
3. 队列可见
4. 队列可取消

### 为什么没有一步到位做成“完整服务端队列”

官方教程背后依赖的是 LangGraph Agent Server 的一整套 run 生命周期接口，例如:

1. 创建后台 run
2. 取消某个 queued run
3. 重新加入某个 run 的 stream

而当前项目的自建后端还没有完整实现这些 run API。  
如果要完全对齐官方服务端方案，下一阶段需要继续补:

1. `/threads/:threadId/runs`
2. `/threads/:threadId/runs/:runId/stream`
3. `/threads/:threadId/runs/:runId/cancel`
4. 后端 run 状态持久化与队列调度

所以这次先落地“项目内可运行的消息队列能力”，保证功能先用起来，再逐步向官方服务端模式收敛。

## 3. 当前已完成的开发

本次已经完成以下接入:

1. `streamStore` 改为 FIFO 队列模型
2. 输入框支持在生成中继续发送，新消息自动入队
3. 停止当前回复时，不会误清空后续排队消息
4. 新增 `QueuePanel` 队列面板
5. 支持单条移除和一键清空
6. 队列项保留创建时间，方便用户识别顺序

## 4. 下一步建议

如果要继续深挖这个子项目，建议按下面顺序推进:

1. 把队列能力补到后端 run API，逐步向官方 `enqueue` 模式靠拢
2. 为队列项增加失败状态与失败重试
3. 队列过长时增加上限提醒
4. 给队列消费过程补自动化测试
